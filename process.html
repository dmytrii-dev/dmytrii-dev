How it works

From repeated workflow to working tool.

The goal is not to automate everything at once. The goal is to define one useful workflow, build a first working version, test it, and improve it around real use.

01

Send the workflow

Describe the process you repeat. Useful materials include screenshots, screen recordings, sample orders, address blocks, exported data, or a short explanation of the systems involved.

02

I check automation fit

I look at whether the workflow is practical to automate, where the data comes from, what can be made reliable, and which parts should stay under user control.

03

We agree the scope

Before building, we define what the first version should do, what it will not do, what inputs it expects, what access or test data is needed, and what counts as delivery.

04

Price and payment are confirmed

Most custom work is handled as a fixed-scope project or split into milestones. Work starts after the scope, price, and payment structure are agreed.

05

First working version

I build the first version around the agreed workflow. For browser-based systems, this may be an extension, internal tool, script, or small workflow interface.

06

Test, refine, deliver

The tool is tested against real examples or a test environment. I refine behavior, fix issues inside the agreed scope, and deliver usage instructions with the final version.

Project materials

What I may need from you.

Automation is easier and safer when the workflow is visible. The exact materials depend on the project, but these are the most useful.

Sample data

Example orders, address blocks, messages, exports, or copied text that represent the real workflow.

Screenshots

Pages, forms, buttons, fields, and the final result the automation should prepare.

Screen recording

A short recording of the manual process often explains the workflow faster than a long document.

Test access

When possible, a test account or safe environment helps reduce blind testing and speed up delivery.

If testing depends on your environment, delivery timing depends on feedback rounds. Clear examples and timely testing make the project faster and more reliable.

Scope and delivery

Small enough to finish, useful enough to matter.

I prefer a clear first version instead of a vague endless project. Improvements can always be added later once the first workflow works.

Included in the first version

  • One clearly defined workflow.
  • Expected inputs, fields, and actions.
  • Testing against agreed examples.
  • Fixes for issues inside the agreed scope.
  • Basic usage instructions after delivery.

Handled as follow-up work

  • New websites, tools, or workflow branches.
  • Changed input formats or new product types.
  • Extra automation rules discovered later.
  • Long-term maintenance or larger system expansion.
